# Contract review methodology

Use this adapted Contract Review Pro workflow for general contract review. The goal is a practical pre-signature risk report, not a formal legal opinion or redline.

## Workflow

1. Build review-state: file name, inferred contract type, represented side if known, extracted text length, review depth, and gate list.
2. Read for transaction shape: parties, subject matter, price/payment, delivery or performance, acceptance, breach/remedies, termination, security or guarantees, confidentiality/IP/data, dispute resolution, attachments, and signatures.
3. Check validity first. Prioritize obvious formation, authority, legality, required approvals, forbidden transaction structures, missing essentials, and signature gaps before clause optimization.
4. Run gates in order: validity, subject/authority, clause coverage, consistency, output.
5. Review clauses with the dual-sided method:
   - Positive: what each party must do in normal performance.
   - Negative: what happens if they do not perform.
   - Enforcement: what happens if the remedy is not honored.
6. For each material risk, include six dimensions: classification, exposure, likelihood, avoidability, business tradeoff, urgency.
7. End with negotiation points and signing tradeoffs. Do not tell the user to sign or not sign.

## Contract type scan

Infer type from the document text where possible:

- Sales or purchase agreement
- Lease agreement
- Loan agreement
- Construction agreement
- Services agreement
- Employment or contractor agreement
- Equity, investment, or shareholder agreement
- Guarantee, security, pledge, or collateral agreement
- Technology, license, or IP agreement
- General contract when the type is unclear

If the type is uncertain, say so and still run the general gates.

## Risk labels

- Validity/compliance
- Subject/authority
- Price/payment
- Delivery/acceptance
- Breach/remedies
- Termination/exit
- Collateral/guarantee
- Liability/indemnity
- IP/confidentiality
- Data/privacy/security
- Dispute resolution
- Definitions/attachments
- Text consistency
- Notices/signatures
- Formatting/mechanics

## Revision routing

Use direct-change language only for objective, low-dispute fixes: typos, dates, notice mechanics, signature mechanics, missing standard notices, or clerical consistency fixes.

Use comment/negotiation language for business tradeoffs, major risk allocation, factual gaps, terms likely to be resisted, or multiple reasonable options.

Four-question check:

1. Can the user directly change it without making a business decision?
2. Does it involve a commercial tradeoff?
3. Would the other side likely accept it?
4. Are there multiple reasonable options?
